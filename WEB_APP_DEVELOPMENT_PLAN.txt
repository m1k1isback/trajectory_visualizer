ПЛАН РАЗРАБОТКИ ВЕБ-ПРИЛОЖЕНИЯ TRAJECTORYVISUALIZER
====================================================

Назначение документа
--------------------

Этот документ предназначен для студента или разработчика, который будет
строить пользовательский веб-интерфейс TrajectoryVisualizer поверх уже
подготовленного вычислительного ядра Core.

Цель будущего интерфейса - получить инженерное приложение, похожее на макет:

- верхнее меню и панель инструментов;
- инспектор окон;
- дерево загруженных данных;
- панель свойств файла;
- консоль сообщений;
- несколько графиков;
- таблица данных;
- 3D-сцена с Землей, траекториями, орбитами, маркерами;
- панели свойств графика и объекта сцены;
- список объектов сцены;
- нижняя панель анимации;
- статусная строка проекта.

Важно: Core не должен превращаться в веб-приложение. Веб-часть должна
использовать Core как обычную Python-библиотеку.


1. Общая архитектура приложения
-------------------------------

Приложение следует строить слоями:

1. Core
   Уже реализован. Отвечает за данные, проекты, статистику, валидацию,
   описания графиков, таблиц, 3D-объектов и анимаций.

2. Backend API
   Python-слой, который принимает запросы от браузера, вызывает Core и
   возвращает JSON.

3. Frontend
   Веб-интерфейс в браузере: окна, панели, графики, таблицы, 3D-сцена,
   формы свойств, меню, статусная строка.

4. Static assets
   Локальные JavaScript/CSS-библиотеки, иконки, Cesium/Plotly или другие
   библиотеки визуализации. По условиям проекта все библиотеки должны лежать
   локально.

Главное правило:

    Frontend не читает расчетные файлы напрямую.
    Backend не парсит расчетные файлы вручную.
    Все расчетные данные проходят через Core.DataLoader.


2. Рекомендуемая структура ответственности
------------------------------------------

Core:

- загружает файлы;
- хранит Dataset;
- валидирует данные;
- считает статистику;
- хранит описания графиков;
- хранит описания таблиц;
- хранит описания 3D-объектов;
- хранит описания анимаций;
- хранит проект и workspace;
- пишет сообщения через Logger.

Backend:

- принимает HTTP-запросы;
- вызывает Core;
- хранит экземпляры менеджеров Core на время работы приложения;
- преобразует объекты Core в JSON;
- принимает команды от Frontend;
- сохраняет и открывает проекты;
- отдает статические файлы интерфейса.

Frontend:

- показывает окна и панели;
- рисует графики;
- рисует таблицы;
- рисует 3D-сцену;
- показывает сообщения;
- отправляет команды на Backend;
- хранит только состояние интерфейса: активное окно, выбранный объект,
  размеры панелей, открытые вкладки.


3. Минимальная серверная часть
------------------------------

Нужно выбрать backend-технологию. Так как Core не зависит от веб-фреймворков,
веб-слой может использовать FastAPI, Flask или другой фреймворк уже вне Core.

Рекомендуемая логика запуска backend:

1. Создать объект состояния приложения.
2. В нем создать экземпляры менеджеров Core:

    DataLoader
    DatasetManager
    VariableManager
    GraphManager
    TableManager
    CesiumManager
    AnimationManager
    ExportManager
    ProjectManager
    Statistics
    DatasetValidator
    Logger

3. Зарегистрировать API-маршруты.
4. Отдать браузеру frontend.

Пример внутреннего состояния backend:

    class AppState:
        def __init__(self):
            self.loader = DataLoader()
            self.datasets = DatasetManager()
            self.variables = VariableManager()
            self.graphs = GraphManager()
            self.tables = TableManager()
            self.scene = CesiumManager()
            self.animations = AnimationManager()
            self.exports = ExportManager()
            self.projects = ProjectManager()
            self.validator = DatasetValidator()
            self.statistics = Statistics()
            self.logger = Logger()

Это состояние не должно находиться в Core. Оно относится к веб-приложению.


4. API для файлов и данных
--------------------------

Нужно реализовать endpoints для работы с файлами.

Минимальные операции:

1. Загрузить файл:

    POST /api/files/load

    Вход:
        путь к файлу или загруженный файл

    Действия:
        dataset = DataLoader.load(path)
        dataset_id = DatasetManager.register(dataset)
        VariableManager.register_dataset(dataset)
        validation = DatasetValidator.validate(dataset)
        stats = Statistics.calculate(dataset)

    Выход:
        dataset_id
        имя Dataset
        список столбцов
        список переменных
        единицы измерения
        количество строк
        количество столбцов
        сообщения валидатора
        базовая статистика

2. Получить список Dataset:

    GET /api/datasets

3. Получить Dataset по id:

    GET /api/datasets/{dataset_id}

4. Получить столбцы Dataset:

    GET /api/datasets/{dataset_id}/columns

5. Получить часть данных для таблицы:

    GET /api/datasets/{dataset_id}/rows?offset=0&limit=100

Важно: для больших файлов нельзя сразу отправлять все строки в браузер.
Таблица должна получать данные порциями.

6. Получить данные для графика:

    GET /api/datasets/{dataset_id}/series?x=time&y=speed

Для графиков можно дополнительно реализовать прореживание данных на backend
или frontend, если точек очень много.


5. Дерево данных слева
----------------------

На макете слева есть дерево:

    trajectory1.txt
        time
        x_ecef
        y_ecef
        z_ecef
        vx
        vy
        vz
        speed
        height
        lat
        lon

Это дерево строится из:

- DatasetManager.list()
- Dataset.column_names
- Dataset.variable_names
- Dataset.units
- VariableManager.list()

Frontend должен:

- показывать каждый Dataset как корневой узел;
- показывать переменные как дочерние узлы;
- показывать единицы измерения, если они есть;
- позволять выбрать переменную;
- позволять перетащить переменную на график или таблицу.

Backend должен отдавать структуру примерно такого вида:

    [
        {
            "dataset_id": "...",
            "name": "trajectory1",
            "argument": "time",
            "columns": [
                {"name": "time", "unit": "s", "is_argument": true},
                {"name": "speed", "unit": "m/s", "is_argument": false}
            ]
        }
    ]


6. Панель свойств файла
-----------------------

На макете есть блок "Свойства файла".

Его можно построить из Dataset, Statistics и ValidationResult.

Показывать:

- имя файла;
- количество строк;
- количество столбцов;
- размер файла;
- аргумент X;
- минимальное значение аргумента;
- максимальное значение аргумента;
- единицы аргумента;
- средний шаг аргумента;
- кодировку;
- разделитель;
- предупреждения валидатора.

Часть данных уже есть:

- dataset.row_count;
- dataset.column_count;
- dataset.argument_name;
- dataset.units;
- dataset.metadata;
- Statistics.calculate(dataset).

Средний шаг аргумента можно считать в backend:

    argument = dataset.argument_values
    average_step = mean(diff(argument))

Но это не нужно добавлять в DataLoader. Это вспомогательная расчетная
характеристика для интерфейса.


7. Графики
----------

На макете есть несколько графиков:

- "График: Скорость";
- "График: Высота";
- "График: Углы".

В Core за это отвечает:

- GraphManager;
- GraphDescription;
- CurveDescription.

Frontend должен рисовать графики выбранной библиотекой. Например:

- Plotly;

- ECharts;
- uPlot;
- Chart.js;
- другая локальная библиотека.

Core не должен импортировать эту библиотеку.

Минимальные API:

1. Создать график:

    POST /api/graphs

    Вход:
        title
        curves
        axes

2. Получить список графиков:

    GET /api/graphs

3. Получить описание графика:

    GET /api/graphs/{graph_id}

4. Обновить свойства графика:

    PATCH /api/graphs/{graph_id}

5. Удалить график:

    DELETE /api/graphs/{graph_id}

6. Получить данные для кривой:

    GET /api/graphs/{graph_id}/data

Описание кривой должно хранить:

- dataset_id;
- x_variable;
- y_variable;
- color;
- legend;
- axis;
- line style;
- line width;
- visibility.

В текущем Core часть этих свойств уже есть явно, часть можно хранить в
словарях. В будущем лучше сделать строгие dataclass-модели для настроек
графика и кривой.


8. Панель свойств графика
-------------------------

На макете справа есть свойства графика:

- название окна;
- название X;
- название Y;
- показывать легенду;
- показывать сетку;
- автомасштаб;

- цвет фона;
- цвет сетки;
- список кривых;

Frontend показывает форму.
Backend принимает изменения и обновляет GraphDescription.

Рекомендуемая модель для будущего расширения:

    GraphStyle:
        show_legend
        show_grid
        auto_scale
        line_style
        line_width
        font_size
        background_color

Эту модель можно добавить позже в Core, если настройки графиков станут
стабильной частью проекта.


9. Таблица данных
-----------------

На макете есть таблица данных с большим количеством строк.

В Core за описание таблицы отвечает:

- TableManager;
- TableDescription.

Dataset хранит сами данные.

Frontend должен:

- показывать только видимую часть таблицы;
- использовать виртуализацию строк, если данных много;
- поддерживать горизонтальную прокрутку;
- показывать заголовки;
- показывать единицы измерения;
- позволять выбрать строки и столбцы.

Backend API:

    GET /api/tables
    POST /api/tables
    GET /api/tables/{table_id}
    PATCH /api/tables/{table_id}
    DELETE /api/tables/{table_id}
    GET /api/tables/{table_id}/rows?offset=0&limit=200

Важно:

    Не отправлять всю таблицу сразу, если строк много.


10. 3D-сцена
------------

На макете есть окно "Cesium 3D" с Землей, траекториями, орбитами и маркерами.

В Core за описание сцены отвечает:

- CesiumManager;
- SceneObjectDescription.

Несмотря на название CesiumManager, Core не использует Cesium. Он только
хранит описание объектов.

Frontend должен:

- создать реальную 3D-сцену;
- загрузить локальную библиотеку Cesium или другую 3D-библиотеку;
- получить из backend список объектов сцены;
- запросить координаты из Dataset;
- построить линии, точки, маркеры.

Backend API:

    GET /api/scene/objects
    POST /api/scene/objects
    GET /api/scene/objects/{object_id}
    PATCH /api/scene/objects/{object_id}
    DELETE /api/scene/objects/{object_id}
    GET /api/scene/objects/{object_id}/coordinates

Описание объекта сцены:

    {
        "object_type": "trajectory",
        "name": "trajectory1",
        "dataset_id": "...",
        "properties": {
            "x": "x_ecef",
            "y": "y_ecef",
            "z": "z_ecef",
            "color": "#00ff00",
            "line_width": 2,
            "show_points": false,
            "show_trace": true
        }
    }

В будущем стоит типизировать свойства для разных object_type:

- trajectory;
- orbit;
- point;
- marker;
- polyline.


11. Панель свойств объекта сцены
--------------------------------

На макете справа есть "Свойства объекта":

- тип объекта;
- имя;
- источник данных;
- координаты X/Y/Z;
- цвет;
- толщина линии;
- показывать точки;
- показывать след.

Frontend показывает форму.
Backend обновляет SceneObjectDescription.

Для выпадающих списков X/Y/Z нужно использовать список переменных Dataset.

Проверки:

- выбранный Dataset существует;
- выбранные переменные существуют;
- координатные столбцы числовые;
- у объекта есть имя;
- цвет имеет корректный формат.

Эти проверки можно делать на backend. Если проверки становятся общими и
важными для проекта, позже их можно вынести в Core.


12. Список объектов сцены
-------------------------

На макете справа снизу есть "Объекты на сцене".

Он строится из:

    CesiumManager.list()

Frontend должен позволять:

- включать и выключать видимость объекта;
- выбирать объект;
- менять цвет;
- удалять объект;
- добавлять маркер;
- добавлять траекторию.

Видимость можно хранить в SceneObjectDescription.properties:

    "visible": true


13. Анимация и timeline
-----------------------

На макете внизу 3D-окна есть:

- play/pause;
- скорость;
- текущее время;
- общий интервал времени;
- ползунок;
- следовать объекту;
- показывать след.

В Core есть:

- AnimationManager;
- AnimationDescription;
- PlaybackState.

Frontend отвечает за реальное воспроизведение.

Backend хранит описание:

- speed;
- current_time;
- mode;
- state.

API:

    GET /api/animations
    POST /api/animations
    PATCH /api/animations/{animation_id}
    DELETE /api/animations/{animation_id}

Для плавной анимации лучше не отправлять запрос на backend на каждый кадр.
Backend хранит состояние, а frontend выполняет локальное проигрывание.

17. Консоль сообщений

Для веб-приложения можно сделать backend-адаптер:

И передавать его в UI через API:

    DELETE /api/logs

Также в консоль нужно отправлять:

- ошибки загрузки;
- предупреждения валидатора;
- создание графика;

19. Сохранение проекта
----------------------

В Core есть ProjectManager и Workspace.

Проект должен сохранять:

- список Dataset или ссылки на исходные файлы;
- описания графиков;
- описания таблиц;
- описания объектов 3D-сцены;
- описания анимаций;
- настройки;
- workspace;
- возможно, layout окон.

Важный момент:

    Dataset содержит NumPy-массивы. Не стоит сохранять весь Dataset в JSON
    проекта, если исходный файл доступен. Лучше сохранять путь к исходному
    файлу и при открытии проекта заново загрузить его через DataLoader.

При открытии проекта:

1. Прочитать project.json.
2. Восстановить workspace.
3. Для каждого файла вызвать DataLoader.load().
4. Зарегистрировать Dataset в DatasetManager.
5. Восстановить графики, таблицы, объекты сцены и анимации.
6. Сообщить пользователю о проблемах, если файл не найден.


20. Экспорт
-----------

В Core есть ExportManager и ExportTask.

Пока он хранит только описание будущего экспорта.

Веб-приложение может сначала реализовать простые виды экспорта:

- экспорт таблицы в CSV;
- экспорт графика в PNG;
- экспорт настроек проекта в JSON;
- экспорт отчета в HTML или PDF.

Важно:

    Если экспорт требует Plotly, браузера, HTML или CSS, эта логика должна
    быть вне Core.


21. Настройки приложения
------------------------

В Core есть Config.

Через него можно хранить:

- язык интерфейса;
- тему;
- путь к последнему проекту;
- список последних файлов;
- настройки отображения;
- настройки логирования.

API:

    GET /api/settings
    PATCH /api/settings

Frontend должен применять настройки после загрузки страницы.


22. Плагины
-----------

В Core есть PluginManager.

На первом этапе достаточно:

- показать список обнаруженных плагинов;
- прочитать plugin.json;
- вывести имя, версию, описание.

Не нужно сразу выполнять код плагинов.

API:

    GET /api/plugins
    POST /api/plugins/discover


23. Работа с большими данными
-----------------------------

Для инженерных расчетов файлы могут быть большими.

Правила:

1. Не отправлять весь Dataset в браузер без необходимости.
2. Таблицы должны использовать постраничную загрузку или виртуализацию.
3. Графики должны использовать прореживание или downsampling, если точек
   слишком много.
4. 3D-сцена должна получать только нужные координатные столбцы.
5. Backend должен возвращать данные в компактном JSON.

Пример API для таблицы:

    GET /api/datasets/{dataset_id}/rows?offset=5000&limit=200

Пример API для графика:

    GET /api/datasets/{dataset_id}/series?x=time&y=speed&max_points=2000


24. Обработка ошибок
--------------------

Ошибки делятся на несколько типов:

1. Ошибки загрузки файла:
   файл не найден, неподдерживаемый формат, нет доступа.

2. Ошибки структуры данных:
   пустой файл, нет заголовков, строки разной длины.

3. Предупреждения данных:
   NaN, Inf, нечисловые значения, немонотонный аргумент.

4. Ошибки действий пользователя:
   выбран несуществующий Dataset, переменная не найдена, нельзя построить
   объект без координат.

Все ошибки должны отображаться:

- в консоли сообщений;
- в уведомлениях;
- рядом с проблемной формой, если ошибка относится к форме.

Не нужно скрывать предупреждения валидатора. Инженер должен видеть, что
происходит с данными.


25. Предлагаемый порядок разработки
-----------------------------------

Этап 1. Backend skeleton

- сделать endpoint /api/health.

Этап 2. Загрузка файлов

- реализовать /api/files/load;

- подключить VariableManager;
- подключить DatasetValidator;
- подключить Statistics;

Этап 3. Дерево данных

- реализовать /api/datasets;
- сделать frontend-панель "Данные";
- показать Dataset и переменные.

Этап 4. Таблица данных

- реализовать постраничную выдачу строк;

- добавить прокрутку;
- добавить заголовки и единицы измерения.

Этап 5. Графики

- реализовать создание GraphDescription;

Этап 6. Панель свойств графика

- сделать выбор активного графика;
- отображать свойства;
- менять цвет, толщину, легенду, видимость кривых;
- сохранять изменения в GraphManager.

Этап 7. 3D-сцена

- подключить локальный Cesium или другую 3D-библиотеку;
- реализовать SceneObjectDescription;
- построить траекторию по x/y/z;
- добавить список объектов сцены;

Этап 8. Анимация

- добавить AnimationDescription;

- сделать ползунок времени;

Этап 9. Проекты

- реализовать создание, открытие, сохранение проекта;

Этап 10. Окна и layout

- реализовать сохранение раскладки;

Этап 11. Консоль и статусная строка

Этап 12. Полировка

26. Что не делать на раннем этапе
---------------------------------

Не стоит сразу делать:

- сложную систему плагинов; ????????????????

- полноценный экспорт PDF;

- продвинутый редактор layout; ???????????

- сложную систему прав пользователей;
- сетевую синхронизацию;
- базу данных, если проект должен быть автономным;
- изменение Core ради удобства конкретной кнопки интерфейса.

Сначала нужно добиться стабильного цикла:

    загрузить файл -> увидеть данные -> построить график -> открыть таблицу
    -> построить 3D-траекторию -> сохранить проект -> открыть проект.


27. Критерии готовности первой версии
-------------------------------------

Первая рабочая версия считается готовой, если:

1. Пользователь может загрузить CSV/TXT/TSV/DAT.
2. Данные появляются в дереве слева.
3. Показываются свойства файла.
4. Предупреждения валидатора видны в консоли.
5. Можно открыть таблицу Dataset.
6. Можно построить минимум один график.
7. Можно построить 3D-траекторию по выбранным X/Y/Z.
8. Можно запустить простую анимацию по времени.
9. Можно сохранить проект.
10. Можно открыть сохраненный проект.
11. Приложение работает без интернета.
12. Core остается независимым от веб-технологий.


28. Итоговая цель
-----------------

Нужно получить веб-приложение, в котором Core является надежным инженерным
ядром, backend является мостом между Core и браузером, а frontend отвечает
только за удобное отображение и управление.

Если студент будет придерживаться этого плана, приложение постепенно придет
к интерфейсу, похожему на присланный макет, без разрушения архитектуры Core.
